👤 사용자 접속 경로

사용자가 브라우저로 서비스에 접속하는 두 가지 경로입니다.

외부 접속: 사용자 → Cloudflare(DDoS 방어) → Tunnel(HTTPS) → Docker → Express:3000 로컬 접속: 사용자 → http://localhost:3000 → Express → SQLite

설명

  • 외부 접속: Cloudflare Tunnel이 공개 HTTPS URL을 생성하여 로컬 서버를 외부에 노출
  • 로컬 접속: 같은 PC/네트워크에서 localhost:3000으로 직접 접속
  • 두 경로 모두 동일한 Express 서버로 도달

🌐 네트워크 / 배포 구조

┌─── Docker Compose ───────────────────────────┐ │ │ │ ┌─ planner ──────────┐ ┌─ tunnel ────────┐ │ │ │ Node.js + Express │ │ cloudflared │ │ │ │ Port 3000 │←─│ http2 프록시 │ │ │ │ Volume: /data │ │ │ │ │ └─────────────────────┘ └────────┬────────┘ │ │ │ │ └────────────────────────────────────│──────────┘ ↓ Cloudflare 네트워크 ↓ 외부 사용자 (HTTPS)

컨테이너 구성

  • planner: Node 20 Alpine 기반, Express 서버 실행, DB를 Docker 볼륨에 저장
  • tunnel: cloudflared 이미지, planner:3000으로 HTTP/2 프록시
  • Docker 볼륨 planner-data로 DB 영속성 보장

서버 바인딩 (v6.1)

  • Docker 환경 (DB_DIR=/data): 0.0.0.0에 바인딩 — tunnel 컨테이너가 Docker 내부 네트워크(172.18.0.x)로 접근 가능
  • 로컬 환경: 127.0.0.1에 바인딩 — 외부 접근 차단 (보안)
  • BIND_HOST 환경변수로 직접 오버라이드 가능
  • Docker에서 0.0.0.0은 격리된 가상 네트워크 내부에서만 노출 → 보안 안전

🛡️ Express 미들웨어 체인

모든 요청은 다음 미들웨어를 순서대로 통과합니다.

요청 수신 ↓ Helmet ─────── HTTP 보안 헤더 자동 설정 (XSS, Clickjacking 방지) ↓ Morgan ─────── 요청 로깅 (combined 포맷, 디버깅용) ↓ Rate Limit ─── 전역: 100req/60초, 로그인: 10req/15분 (브루트포스 차단) ↓ Body Parser ── JSON + text/plain 파싱 (sendBeacon 지원) ↓ Session ────── express-session + SQLite 세션 스토어 (7일 유지, 15분 정리) ↓ 라우트 핸들러

각 미들웨어 역할

  • Helmet: Content-Security-Policy, X-Frame-Options 등 보안 헤더 자동 추가
  • Morgan: 모든 HTTP 요청을 로그에 기록 (IP, 메서드, 경로, 상태코드)
  • Rate Limit: 짧은 시간에 과다 요청 차단 → DDoS/브루트포스 방지
  • Session: 쿠키 기반 세션으로 로그인 상태 유지, sessions.db에 저장

관련 파일

🔐 인증 흐름

login.html (로그인 폼) ↓ POST /api/auth/login (username, password) ↓ 서버: findUser(username) → DB에서 유저 조회 ↓ bcrypt.compare(password, hash) → 비밀번호 해시 비교 ↓ (일치) req.session.regenerate() → 새 세션 ID 발급 (세션 고정 공격 방지) ↓ req.session.userId = user.id → 세션에 유저 ID 저장 ↓ 응답: { username } → 브라우저가 / 로 리다이렉트 ↓ 이후 모든 요청: requireAuth() 미들웨어가 session.userId 검증 ↓ (없으면) 401 응답 또는 login.html로 리다이렉트

보안 포인트

  • bcrypt: 비밀번호를 해시로 저장 → 원문 복구 불가, 레인보우 테이블 공격 무력화
  • session.regenerate(): 로그인 성공 시 세션 ID 교체 → 세션 고정 공격 차단
  • httpOnly 쿠키: JavaScript에서 세션 쿠키 접근 불가 → XSS로 세션 탈취 방지
  • 로그인 Rate Limit: 15분에 10회로 제한 → 무차별 대입 공격 차단

  • login.html — 로그인 폼 UI

🗄️ SQLite 데이터베이스

DB 파일

  • planner.db: 유저 정보 + 플래너 데이터 (Docker: /data/, 로컬: DB_DIR 경로)
  • sessions.db: 세션 데이터 (같은 디렉토리)

스키마

users ├─ id INTEGER PRIMARY KEY AUTOINCREMENT ├─ username TEXT UNIQUE NOT NULL ├─ password_hash TEXT NOT NULL ├─ is_admin INTEGER DEFAULT 0 └─ created_at DATETIME DEFAULT CURRENT_TIMESTAMP user_data ├─ user_id INTEGER PRIMARY KEY → users.id (FK) ├─ data TEXT DEFAULT '{}' (JSON blob, 최대 1MB) └─ updated_at DATETIME DEFAULT CURRENT_TIMESTAMP

JSON blob 구조 (user_data.data)

{ sections: [...] // 체크리스트 섹션 (반복 일정) completions: {} // 날짜별 체크 완료 상태 memos: {} // 날짜별 메모 mealPlans: {} // 날짜별 식단 계획 mealChecks: {} // 식단 체크 (레거시) mealTimes: {} // 끼니별 시간 weights: {} // 날짜별 몸무게 targetWeight: number // 목표 몸무게 dailyRecords: {} // 일일 스냅샷 (tracker용) } 날짜 키 형식: "YYYY-M-D" (예: "2026-4-7")

특징

  • WAL 모드: 읽기/쓰기 동시 가능 → 성능 향상
  • Foreign Key: user_data.user_id → users.id 참조 무결성
  • Upsert 패턴: INSERT ON CONFLICT UPDATE → 데이터 없으면 생성, 있으면 갱신

관련 파일

  • db.js — DB 초기화, 스키마, CRUD 함수
  • scripts/create-user.js — CLI 유저 생성 도구
better-sqlite3 WAL mode JSON blob

🔄 데이터 동기화 흐름

── 데이터 로드 (페이지 시작) ── GET /api/auth/me → 유저명 확인 (localStorage 키 분리용) GET /api/data → 서버에서 JSON 로드 ↓ 실패 시 localStorage.getItem('planner_v4_{username}') → 오프라인 캐시 폴백 ── 데이터 저장 (유저 조작 시) ── data 객체 수정 ↓ saveData() 호출 ├─ localStorage.setItem() ──── 즉시 (오프라인 캐시) └─ fetch PUT /api/data ─────── 300ms 디바운스 (서버 저장) ↓ 실패 시 화면에 경고 표시 ── 페이지 닫기 ── navigator.sendBeacon(POST /api/data) → 마지막 데이터 전송 (비동기, 차단 없음) ── 전체 흐름 요약 ── loadData() → 메모리(data 객체) → saveData() → localStorage(즉시) + debounce(300ms) → 서버 ↘ 페이지 닫기: sendBeacon

핵심 개념

  • 서버가 원본 (Source of Truth): 페이지 로드 시 항상 서버 데이터 우선
  • localStorage는 캐시: 서버 장애 시에만 사용하는 오프라인 폴백
  • 300ms 디바운스: 빈번한 조작 시 서버 요청을 묶어서 전송 (성능 최적화)
  • sendBeacon: 탭/브라우저 닫아도 데이터 손실 없이 마지막 저장 보장
  • 충돌 해결: Last Write Wins (마지막 저장이 이김, 머지 로직 없음)

모든 모듈은 전역 스코프에서 함수를 정의하며, 로드 순서가 의존성을 결정합니다.

의존성 상세

모듈함수 수의존 대상역할
data.js60+없음 (기반 레이어)전역 data 객체, 모든 CRUD, 서버 동기화, escapeHtml

로드 순서가 중요한 이유

  • ES Module이 아닌 전역 스코프 방식 — <script> 순서가 곧 의존성 해결 순서
  • data.js가 먼저 로드되어야 다른 모듈이 data 객체와 CRUD 함수에 접근 가능
  • checklist.js가 calendar.js/tracker.js보다 먼저 로드되어야 syncCalendar() 등 함수 사용 가능
  • 순서: data.js → checklist.js → calendar.js → tracker.js
